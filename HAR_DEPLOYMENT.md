# STM32 Cube AI HAR deployment notes

This project now contains an application skeleton for a Human Activity
Recognition (HAR) system:

- Input: UART, SD card, USB, Ethernet
- Inference: X-CUBE-AI entry point through `MX_X_CUBE_AI_Process()`
- Output: serial text and optional touch-screen display hook

## Current project status

The current CubeMX/MDK project targets `STM32F767IGTx` and uses `stm32f7xx`
HAL files. This matches the confirmed STM32F767IGT6 target.

The generated `.ioc` currently enables only GPIO and X-CUBE-AI. UART, SD,
USB, Ethernet, FATFS, LwIP, and display drivers still need to be enabled in
CubeMX for real hardware I/O. No development board is connected yet, so the
current project is prepared as a software-side integration skeleton.

## Training workflow

Cube AI does not train the model on the STM32. Train the HAR model in Python,
export it, and then import the trained model into X-CUBE-AI:

1. Collect labeled motion data as CSV.
2. Train with `training/train_har.py`.
3. Export `har_model.keras` and `har_model.tflite`.
4. Import the model in STM32CubeMX X-CUBE-AI.
5. Generate the Cube AI network C files.
6. Bridge `HAR_RunNetwork()` to the generated `ai_network_run()` call.

## Data format

Each input channel should provide one complete ASCII line per sensor sample:

```text
ax,ay,az,gx,gy,gz
timestamp,ax,ay,az,gx,gy,gz
```

Acceleration units should match model training, typically `g`. Gyroscope units
should also match training, typically `dps` or `rad/s`.

The default window is:

- `HAR_WINDOW_SAMPLES`: 128 samples
- `HAR_WINDOW_STRIDE`: 64 samples
- `HAR_AXIS_COUNT`: 6
- flattened AI input order: `[sample0_ax, sample0_ay, ... sample127_gz]`

Change these constants in `Inc/har_app.h` if your trained model uses a
different input shape.

## CubeMX configuration checklist

Enable the required hardware in CubeMX and regenerate code:

1. USART/UART for command/data input and serial result output.
2. SDMMC + FATFS for reading CSV files from SD card.
3. USB Device CDC or USB Host, depending on whether the board receives data as
   a virtual COM device or from a USB storage/device source.
4. Ethernet + LwIP for UDP/TCP sample input.
5. LTDC/DSI/FMC/TouchGFX or the board display stack if touch-screen output is
   required.
6. X-CUBE-AI with the trained HAR model imported and code generated.

After regeneration, keep the user-code sections and re-add `Src/har_app.c` to
the MDK project if CubeMX removes manually added files.

## Hardware integration hooks

`Src/har_app.c` defines weak hooks. Implement the needed hooks in a separate
board file, for example `Src/har_board_io.c`, after CubeMX creates the related
drivers:

```c
#include "har_app.h"
#include "usart.h"

int HAR_UART_ReadLine(char *buffer, size_t buffer_size)
{
  /* Copy one complete UART line into buffer and return 1.
     Return 0 when no complete line is available. */
  return 0;
}

void HAR_OutputSerial(const char *text)
{
  HAL_UART_Transmit(&huart1, (uint8_t *)text, strlen(text), 100);
}
```

Required read hooks:

- `HAR_UART_ReadLine`
- `HAR_SD_ReadLine`
- `HAR_USB_ReadLine`
- `HAR_ETH_ReadLine`

Required output hooks:

- `HAR_OutputSerial`
- `HAR_OutputDisplay`

The application polls all input hooks from `HAR_Process()`. Keep hooks
non-blocking so the main loop remains responsive.

## X-CUBE-AI model bridge

The weak fallback `HAR_RunNetwork()` is only a heuristic placeholder so the
application layer has a complete call path before a real model is generated.
Replace it with an implementation that calls the X-CUBE-AI network API.

Typical structure:

```c
#include "har_app.h"
#include "network.h"
#include "network_data.h"

static ai_handle network = AI_HANDLE_NULL;
static ai_u8 activations[AI_NETWORK_DATA_ACTIVATIONS_SIZE];

int HAR_RunNetwork(const float *input,
                   size_t input_element_count,
                   float *output_scores,
                   size_t output_class_count)
{
  ai_buffer ai_input[AI_NETWORK_IN_NUM] = AI_NETWORK_IN;
  ai_buffer ai_output[AI_NETWORK_OUT_NUM] = AI_NETWORK_OUT;

  (void)input_element_count;
  (void)output_class_count;

  ai_input[0].data = AI_HANDLE_PTR(input);
  ai_output[0].data = AI_HANDLE_PTR(output_scores);

  return ai_network_run(network, ai_input, ai_output) == 1;
}
```

Initialize `network` once during `MX_X_CUBE_AI_Init()` or from a helper called
inside `HAR_Init()`, using the exact init sequence generated by your X-CUBE-AI
version.

## Result output

Serial output format:

```text
HAR,source=UART,label=walking,confidence=0.680
```

Class labels are defined in `Src/har_app.c`:

- walking
- running
- sitting
- standing
- upstairs
- downstairs
- unknown

Adjust the order to match the model output tensor class order.
